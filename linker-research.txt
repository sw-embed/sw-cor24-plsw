PL/SW currently goes plsw -> .s -> as24 -> .bin

there is no true binary linker yet

you can concatenate .s files, but local labels collide

you want to split SNOBOL4 into modules like lexer, parser, utils, runtime, main loop

The cleanest path is:

Recommendation
Build a very small static linker around a relocatable object format, and make the compiler/assembler emit:

module-local labels

exported symbols

imported symbols

relocation records

optional library export table metadata

That gives you:

separate compilation

static linking of modules into one executable

optional runtime lookup tables for plugin-ish dispatch

no need for ELF, COFF, dynamic loader, GOT/PLT, etc.

The core problem: label collisions
Your immediate .s concatenation problem is not really a linker problem first — it is also a label namespace problem.

Minimum compiler/assembler change
Every label that is not exported should be made module-unique.

For example, instead of:

L1:
L2:
emit:

$mod_parser$L1:
$mod_parser$L2:
or

parser__L1:
parser__L2:
This alone will let you concatenate .s files more safely.

But it is not enough for real modularity, because you still need:

calls across modules

data references across modules

a way to resolve addresses after layout

So namespacing fixes duplication, but you still need relocations for a real solution.

Two implementation approaches
Approach A — “Poor man’s linker” at assembly level
Idea
Keep everything as text assembly longer.

Each module compiles to .s

exports and imports are represented with directives

a simple ln24 tool reads all .s files

renames locals

collects exports

resolves imports

concatenates sections

patches symbolic references

emits one final .s

as24 then converts that to final .bin

Pros
easiest to debug

leverages your existing assembler

human-readable

good for bring-up

Cons
linker must understand enough assembly syntax to patch references

brittle if assembly syntax evolves

harder to support arbitrary expressions cleanly

less future-proof

This is a good first milestone.

Approach B — true relocatable .o24 + tiny linker
Idea
Teach as24 to emit a compact relocatable object format.

compiler still emits .s

assembler emits .o24

linker works on bytes + symbols + relocations

final output is absolute .bin

Pros
cleaner long-term

easier to add archives/libraries later

easier to support relocation correctly

much closer to standard toolchain architecture

Cons
more work up front than assembly-text linking

requires object format design

This is what I recommend, even if the format is tiny and custom.

What compiler changes are needed?
Your compiler needs to distinguish several symbol classes.

1. Local/internal procedures and labels
These must not be visible outside the module.

Compiler should emit them with unique names, for example:

parser__scan_token
parser__L1
parser__L2
or maintain a module id in metadata.

2. Exported procedures/data
The compiler needs syntax or conventions for “public” symbols.

For example in PL/SW you may want something like:

EXPORT PROC SCAN_TOKEN;
EXPORT DCL TOKEN_TABLE PTR;
This should result in symbol table entries marked global/exported.

3. Imported procedures/data
The compiler must know that some references are external:

EXTERN PROC EMIT_ERROR;
EXTERN DCL GLOBAL_FLAGS PTR;
These become undefined symbols in the object file.

4. Section awareness
Compiler/assembler must separate:

code / text

initialized data

uninitialized data

Even if your runtime image is flat, linker needs logical sections.

5. Relocatable expressions only
The compiler should avoid baking in absolute addresses too early.

For example, instead of forcing a fixed address for:

LOAD R0, SOME_EXTERN
the assembler should emit:

placeholder value 0

relocation record: patch this word with absolute address of SOME_EXTERN

What assembler changes are needed?
This is the key.

Your assembler currently sounds like it emits an absolute image with origin 0.
To support linking, it must support relocatable output.

Minimal features for as24
1. Symbol table output
For every symbol:

name

section

offset within section

binding: local/global/undefined

2. Relocation table output
For every place that references a symbol whose final address is unknown:

section containing patch site

offset of patch site

relocation type

referenced symbol

addend if needed

3. Sectioned output
Instead of one flat image, assembler should emit:

.text bytes

.data bytes

.bss size only

4. Undefined symbol support
Assembler must allow this:

.extern emit_error
call emit_error
and not fail immediately; instead it emits relocation against undefined symbol.

5. Optional export/import directives
For example:

.global scan_token
.extern emit_error
.section .text
What relocation types do you need?
Keep this very small.

Probably enough for COR24 v1:
R_COR24_ABS24
Patch a full 24-bit absolute address into an instruction/data word.

Use for:

function pointers

direct absolute calls/jumps if your ISA uses them

addresses of global data

R_COR24_PCREL
Patch a PC-relative displacement.

Use for:

relative branch instructions

R_COR24_DATA24
Same practical idea as ABS24 but for raw data initializers like:

DC24 some_symbol
You may end up implementing ABS24 and DATA24 the same way.

That is enough for an MVP.

What should the linker do?
Your tiny linker ln24 should:

1. Read all .o24 files
and maybe archive files later.

2. Build a global symbol table
every exported symbol inserted

duplicates are an error unless you later support weak/common

undefineds must be resolved by some export

3. Lay out sections
For example:

all .text modules concatenated

then all .data

then .bss assigned after that

You can keep this dead simple.

4. Compute final addresses
For each symbol:

final_addr = section_base + symbol_offset
5. Apply relocations
For every relocation record:

find target symbol final address

compute relocated value

patch bytes

6. Emit final flat binary
with entry point at chosen symbol, maybe _start or main.

How should libraries work?
For your use case, the simplest library model is not a runtime shared library.

Do this instead:

Static library = archive of .o24
Example:

lexer.o24
parser.o24
runtime.o24
string.o24
packed into:

libsnobol.a24
The linker then:

scans main objects first

notes unresolved externs

pulls in only needed objects from archive

repeats until fixed point

That is optional for phase 1.
You can initially just pass all .o24 files directly to the linker.

Runtime procedure tables vs linker symbol resolution
You mentioned:

I just need a way for a main to be pointed at a table at load/run time to find procedure entry points

That is a different layer from linking.

You likely want both:

1. Link-time symbol resolution
Needed so compiled modules can call each other and refer to data.

2. Runtime export tables
Useful for:

interpreter builtins

dispatch by name/opcode

optional library registration

modular initialization

Example runtime table:

struct proc_entry {
  char *name;
  void *entry;
}
In COR24-ish layout terms:

snolib_exports:
  dc24 name_scan_token
  dc24 scan_token
  dc24 name_match
  dc24 match
  dc24 0
  dc24 0
Then main can be passed pointer to this table.

This does not replace a linker. It complements it.

Before linking, make sure your ABI is stable across modules.

Recommended architecture
Use a three-layer model:

Layer 1 — module source/build
Each PL/SW module compiles independently to one .s, then to one flat binary.

Layer 2 — packaged module image
A post-processing step turns that binary into a loadable module blob:

[ raw as24 binary ][ module metadata trailer ]
Layer 3 — runtime loader/dispatcher
A monitor/loader loads each module into arbitrary memory locations, reads metadata, builds a module table, and offers controlled cross-module calls.

Key idea: no direct cross-module machine-code references
This is the most important simplification.

If modules are loaded into non-contiguous memory and you cannot relocate binary code, then modules should not directly embed addresses of other modules.

Instead:

module A does not call module B by absolute address baked into code

module A calls a known monitor/service entry

it passes a module ID + procedure ID or similar

the monitor finds B in the loaded module table and dispatches it

That turns “linking” into runtime dispatch.

This is exactly why your current monitor/shell/editor pattern works well as a foundation.

High-level runtime picture
At runtime, memory looks something like:

000000  monitor
001800  shell module
004200  editor module
007000  parser module
00A000  lexer module
00C000  shared data buffer
...
The monitor owns a table like:

module  name     load_addr   entry_count   exports_ptr
------  -------  ---------   -----------   -----------
1       shell    0x001800    5             ...
2       editor   0x004200    7             ...
3       parser   0x007000    4             ...

Stage 1 — preprocess assembly to namespace labels
This addresses your immediate modular build problem.

Tool: modpp
Purpose
Rewrite labels so every module gets a unique namespace.

Input
parser.s

prefix parser__

Output
parser.pp.s

Rewrite rules
For every non-public label:

L1:         -> parser__L1:
loop_top:   -> parser__loop_top:
And every reference updates accordingly.

Public entry labels
Keep a small set unchanged or rewrite systematically.

Example:

internal labels: prefixed

exported procedures: either left as-is or rewritten to stable names like parser__PARSE

Benefit
You can independently compile modules without concatenation collisions.

Restriction

Stage 2 — define a module ABI
Before packaging, define what a loadable module is.

A module should have:

a module name/id

one module entry point for initialization

zero or more exported procedures

optionally required services

optionally required monitor ABI version

Example exported procedures
For lexer:

INIT

RESET

For parser:

INIT

Important rule
A module may call:

its own local procedures directly

other modules only via monitor dispatch

This rule is what makes non-contiguous loading feasible without relocation.

Stage 3 — package flat binary with metadata trailer
Because as24 output cannot change, package the binary afterward.

Tool: modpkg
Purpose

Input

Suggested packaged module format
Simplest form:

[ raw module code/data bytes ]
[ metadata trailer ]
[ trailer footer with magic + trailer offset ]

Example layout

footer:
  magic = "MD24"
  metadata_offset = <24-bit or 32-bit offset>
The loader:

loads blob

Keep it tiny.

Module header fields

flags

Export entries
Each export should contain:

This is crucial: offset, not absolute address.

Because the loader computes:

absolute_entry = module_load_address + export_offset
That is exactly what you want for non-contiguous memory loading.

Calling convention for cross-module calls
Because modules are independently loaded, define a small ABI.

Recommended ABI
For cross-module calls, do not try to pass many values in registers.

Instead pass an argument block pointer.

Caller provides
proc/module IDs in agreed registers or stack slots

pointer to argument/result block

Callee returns
return code in one register

result pointer or updated argument block

This is much easier to keep stable across modules.

Example arg block
For NEXT_TOKEN:

arg_block:
  source_ptr
  source_len
  cursor_ptr
  token_out_ptr
  rc
Monitor dispatches the callee with pointer to this block.

How a finished binary works when loaded into non-contiguous memory
Now to your most important practical question.

Assume these modules:

monitor at 0x000000

shell at 0x002000

editor at 0x005800

parser at 0x009000

lexer at 0x00D000

Each module binary is assembled as a self-contained internal image whose internal control flow uses:

local branches

local procedure calls

stack/frame relative accesses

No instruction in parser contains “jump to lexer absolute 0x00D123”.

Instead, parser does this:

prepare argument block

invoke monitor service CALL_EXPORT

pass module id = LEXER

pass proc id = NEXT_TOKEN

pass argument block pointer

Monitor computes:

lexer_base + next_token_offset = actual entry address
and transfers control there.

So non-contiguous placement is fine because the only cross-module addresses are computed at runtime from metadata.

What “PIC” means here
Strictly speaking, this is not full machine-level PIC in the ELF sense.

It is a discipline-based relocatable module model:

Allowed inside module
relative/local control flow

internal symbol references that as24 already resolves within that module

data accessed via pointers given at entry

fixed monitor service entrypoints, if those are globally stable

Forbidden inside module
absolute addresses of other modules

embedded pointers to external data buffers unless provided at runtime

hard-coded assumptions about where caller/callee modules were loaded

That is enough.

The one special case: monitor service entrypoints
You likely need a few fixed absolute entrypoints known to all modules.

Example:

MON_CALL_EXPORT

MON_GET_MODULE

MON_ALLOC

MON_PUTS

MON_RETURN

Because monitor is always at 0 or otherwise fixed.

This is the only acceptable absolute-address dependency in the simple design.

That matches your current monitor-centered architecture.

What the compiler should change
Since you want to stay simple, compiler changes should be minimal.

Needed changes
1. Module awareness
Compiler should know a module name so the preprocessor can use it consistently.

2. Export declarations
Compiler should know which procedures are externally callable.

Example conceptually:

EXPORT PROC PARSE;
EXPORT PROC INIT;
3. External procedures become service calls, not direct CALLs
This is the biggest conceptual change.

Instead of:

EXTERN PROC NEXT_TOKEN;
X = NEXT_TOKEN(...);
compile cross-module calls as something like:

call monitor service CALL_EXPORT(module_id, proc_id, arg_block)
or compile through a helper stub.

4. Optional manifest generation
Compiler could emit sidecar text listing exports.

That would help modpkg.

What the preprocessor should do
modpp is now very important.

It should:
prefix internal labels

prefix internal procedure names

leave exported names stable or make them predictable

maybe emit a label map

Example
Input:

PARSE:
L1:
  ...
HELPER:
L2:
Output:

parser__PARSE:
parser__L1:
  ...
parser__HELPER:
parser__L2:
If PARSE is exported, then manifest says:

export PARSE = parser__PARSE
So modpkg can locate that label offset.

The manifest format
Keep it dead simple.

Example parser.mod:

name parser
id 3
abi 1
entry INIT
export 1 INIT
export 2 PARSE
export 3 SET_SOURCE
Where:

1, 2, 3 are proc IDs

symbolic names are for debugging/tools

actual entry offsets come from symbol lookup against assembly/listing/map

If as24 emits a listing or symbol table file, use that.
If not, modpp can emit one.

How to get symbol offsets without changing as24
This matters.

Since as24 output cannot change, one of these must be true:

Option A — as24 already emits a listing/map
Then modpkg reads it.

Option B — preprocessor computes offsets indirectly
Harder, not ideal.

Option C — compiler/preprocessor emits a tiny “symbol inventory”
Then modpkg combines:

symbol names from inventory

final addresses from assembler listing if available

Best case
Use assembler listing/map if it exists. That is the simplest.

If not, the easiest fallback is to make modpp generate an assembly listing-friendly annotation and rely on parsing whatever as24 already prints.

How cross-module imports should work
With this architecture, “imports” do not mean linker externs.

They mean declared runtime dependencies.

For example parser depends on lexer export IDs:

module LEXER = 4

proc NEXT_TOKEN = 2

These can live in:

source declarations

manifest

a shared .msw constants file

Then parser compiles a call to the monitor dispatcher using those IDs.

That is far simpler than binary extern resolution.

Shared data
You also mentioned data buffers loaded elsewhere.

That fits naturally.

Rule
Modules should access shared data through pointers passed in:

at init time

in arg blocks

from monitor service lookup

Do not bake shared buffer addresses into modules unless they are fixed by system convention.

Example
Shell calls editor with:

pointer to shared edit buffer

pointer to file descriptor block

console I/O handles

Editor then operates entirely via passed-in pointers.

That is already close to what you have working.

Benefits and tradeoffs
Benefits
works with unchanged as24

avoids binary relocation

handles non-contiguous memory naturally

close to what you already proved with monitor/shell/editor

easy to debug because monitor sees all cross-module calls

module binaries are independently buildable and loadable

Tradeoffs
cross-module calls are indirect through monitor

slightly slower than direct linked calls

exported ABI must be explicit and stable

modules cannot casually share globals by direct symbol reference

For your system, these tradeoffs are probably excellent.

My recommended minimum viable implementation order
Phase 1 — label-safe modular assembly
Build modpp to prefix internal labels.

Phase 2 — package modules
Build modpkg to append metadata trailer with:

module id

module name

export IDs

export offsets

entry offset

Phase 3 — monitor registry
Extend monitor to:

load module blob

parse trailer

register module

Phase 4 — monitor dispatcher
Add CALL_EXPORT(module_id, proc_id, arg_ptr).

Phase 5 — compiler/runtime convention
Compile inter-module calls as monitor dispatches instead of direct calls.

That is enough to get a modular SNOBOL4 runtime working.

Strong recommendation on IDs
Use numeric IDs for both modules and exports.

For example:

Module IDs
1 = shell

2 = editor

10 = sn_main

11 = sn_lex

12 = sn_parse

Proc IDs
Within sn_lex:

1 = INIT

2 = NEXT_TOKEN

3 = RESET

Keep names only for debugging and manifests.

That makes the runtime very small.

Bottom line
Given your constraints, the simplest robust solution is:

Build-time
compile each module separately

preprocess each .s to prefix local/internal labels

assemble normally with unchanged as24

append a metadata trailer describing module ID, entry point, and export offsets

Runtime
monitor loads each module anywhere in memory

monitor builds a module/export table

modules call other modules only through a monitor dispatch service

shared data is passed by pointer, not by fixed absolute symbol reference

That gives you a practical module system without a linker and works naturally with non-contiguous memory.
