Investigate and fix pointer dereference (->) failing when multiple BASED records are defined across includes (GitHub issue #4). Currently, when two or more BASED records are defined (e.g., DESCR and OBJHDR via separate %INCLUDE files), the -> operator fails with 'undefined field in pointer dereference'. It works with a single BASED record but breaks when a second is added. Investigate the codegen path for ptr->field dereference to understand how BASED record fields are resolved. The likely problem is that the field lookup only searches the most recently defined BASED record instead of all known BASED records. Options: (A) search all BASED records for the field name (simple but ambiguous if two records share a field name), (B) associate PTR variables with a specific BASED type (PL/I-style, more correct but larger change). Implement whichever approach fits the current architecture, add a test with multiple BASED records, and verify the SNOBOL4 use case (descr.msw + heap.msw) would work.
